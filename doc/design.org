#+title: git-branch-off — Design Notes
#+author: baleti

* Architecture

The package is split into eight focused files plus a root entry-point:

| File                          | Responsibility                                             |
|-------------------------------+------------------------------------------------------------|
| =git-branch-off.el=           | Requires all submodules; defines =git-branch-off-setup=    |
| =git-branch-off-stage.el=     | Line/hunk precision staging and amend helpers              |
| =git-branch-off-commit.el=    | Stage-and-commit, stage-and-commit-branch-off              |
| =git-branch-off-log.el=       | Log overlay marking, n/p navigation with live preview      |
| =git-branch-off-worktree.el=  | Create / delete detached worktrees at branch-off commits   |
| =git-branch-off-reword.el=    | Git plumbing layer; reword/remove with branch-off cascade  |
| =git-branch-off-squash.el=    | Mark system, topological chain building, async squash      |
| =git-branch-off-blob.el=      | n/p navigation in magit-blob with branch-off awareness     |
| =git-branch-off-search.el=    | Four consult-based git history search commands             |

All public symbols use the =git-branch-off-= prefix.  Internal symbols use
=git-branch-off--= (double dash).

* The refs/branch-off/ Namespace

Branch-off commits live under =refs/branch-off/<full-40-char-hash>=.  The ref
name encodes the commit hash, making lookups O(1): to check whether commit
=abc123...= is a branch-off, test for the existence of
=refs/branch-off/abc123...=.

This namespace is invisible to the normal branch plumbing:

- =git branch= / =git log= (without =--all=) ignore it
- =git push= / =git fetch= default refspecs ignore it
- =git gc= will prune unreferenced commits, but the ref itself keeps them alive

The hash-as-refname property also makes cascading updates straightforward:
when a branch-off commit's parent is rewritten (via reword/remove/squash),
=git-branch-off--cascade= walks all =refs/branch-off/*= refs, detects those
whose parent changed, creates new commits with the updated parent, updates the
refs, and recurses until no more changes propagate.

* Line-Precision Patch Building (git-branch-off--patch-from-diff)

Standard =git apply --cached= operates at hunk granularity: you stage a whole
hunk or none of it.  This package implements sub-hunk line precision by
building a synthetic patch string from a =git diff -U0= output.

The function =git-branch-off--patch-from-diff= takes:
- DIFF-TEXT :: the raw output of =git diff -U0 -- <file>=
- REL-PATH  :: the file path (used for the patch header)
- SEL-START, SEL-END :: new-file line numbers bounding the selection

It returns =(PATCH-STRING . CHANGE-COUNT)= or nil.

** Algorithm

The parser tracks three cursors across the diff:

- =old-cursor= :: current line in the old file
- =new-cursor= :: current line in the new file
- =context-anchor= :: last old-file line seen before a run of changes
  (used to compute correct hunk headers for non-contiguous selections)

For each line in the diff:

1. *Hunk header* (=@@=) — resets old/new cursors, extracts context-anchor.
2. *Added line* (=+=) — if =new-cursor= is in =[sel-start, sel-end]=, the line
   is accumulated into the current output group.  Otherwise the current group
   is flushed (with any paired deletions).
3. *Deleted line* (=-=) — accumulated into a pending deletion run.  If a
   selected addition follows, the deletions are paired with it to produce a
   replacement hunk.  If a non-selected addition follows, the deletions are
   discarded (the replacement is not being staged).  If a context line follows,
   the deletion is emitted as a standalone deletion if it falls within the
   selection.
4. *Context line* — flushes any open addition group and any pending standalone
   deletions, then advances both cursors.

A running =staged-net-so-far= accumulator tracks the net line count inserted by
previous output hunks so that subsequent hunk headers reference the correct
new-file line numbers in the post-staging index.

** Why -U0?

Using =-U0= (no context lines) simplifies the parser: every line is either a
pure addition, pure deletion, or a hunk header.  Context lines in =-U3= hunks
would require tracking which context lines are inside or outside the selection,
adding significant complexity.  The patch itself is applied with
=--unidiff-zero= to match.

* Cascade Rewriting (git-branch-off--cascade)

When any commit on the ancestry path of branch-off commits is rewritten
(reword, remove, or squash), the branch-off commits that descend from it must
also be rewritten to point to the updated parent.

=git-branch-off--cascade= implements this as a fixpoint iteration:

1. Enumerate all =refs/branch-off/*= refs.
2. For each, read its parent hash via =git cat-file commit=.
3. If the parent is a key in the provided REMAP table (old-hash → new-hash),
   create a new commit object via =git commit-tree= (preserving all author/
   committer metadata), update the ref to the new hash, add the mapping to
   REMAP, and set a =changed= flag.
4. If any refs changed, recurse — some branch-offs may be children of other
   branch-offs that were just rewritten.

Because the ref name encodes the hash, old refs are deleted and new ones are
created rather than moved.

* Squash Algorithm

Squashing N branch-off commits into one is non-trivial because they may not
form a simple linear chain: they might be siblings (same parent), or interleaved
with regular commits.

** Sorting

=git-branch-off--squash-sort-siblings= separates the marked set into:
- A linear chain (each commit's parent is the previous one in the set)
- Siblings (commits sharing a common ancestor outside the set)

Siblings are topologically sorted by ancestry so the resulting squash order
respects the DAG structure.

** Conflict Resolution

When squashing N commits with overlapping edits, merging their trees may
produce conflicts.  =git-branch-off--squash-resolve-path-list= handles this
file-by-file:

1. For each conflicted file, a temporary index is checked out.
2. A dedicated conflict-resolution buffer is opened with the three-way content
   (base / ours / theirs).
3. After the user resolves, the buffer content is written back into the
   temporary index and the next file is processed.
4. Only when all files are resolved does =git-branch-off--squash-finish= create
   the final squashed commit.

* Worktree Placement

Worktrees are created at =<repo-root>/.worktree/<full-40-char-hash>=.  Using the
full hash as the directory name:

- Avoids collisions between branch-off commits at the same logical depth
- Makes the worktree path stable (the commit hash never changes)
- Allows =git worktree list --porcelain= to correlate HEAD hashes with paths
  without any additional bookkeeping

The log overlay code uses =git worktree list --porcelain= to extract all
current HEAD hashes and marks any log line whose hash prefix matches.

* Framework Independence

The package deliberately avoids all doom-specific APIs:

| What to do              | Doom way           | This package uses           |
|-------------------------+--------------------+-----------------------------|
| Load after package      | =after!=           | =with-eval-after-load=      |
| Bind keys in mode-map   | =map!=             | =define-key=, =evil-define-key= |
| Override faces          | =custom-set-faces!= | =defface= with =:extend t=  |
| Query theme colors      | =doom-color=       | not used (face is theme-neutral by default) |

This means the package works unchanged in vanilla Emacs, Spacemacs, and any
other configuration that provides magit (and optionally evil and consult).
