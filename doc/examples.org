#+title: GitQ — Worked Examples
#+author: baleti

Every example below was run against this repository's own history, so you can
reproduce them all here.  Each works identically in two forms:

- *Emacs*: =M-x gitq=, then type the pipeline (live completion per token,
  live preview of results while you type; RET runs it for real).
- *CLI*: =bin/gitq 'PIPELINE'= from anywhere inside the repo.

Sample outputs were captured at commit =99f9179= — counts and hashes will
have moved on, the shapes won't.  In Emacs, results land in the =*gitq*=
buffer; the CLI prints the same frames as plain text.

* Basics

#+begin_example
  $ gitq 'commits /count'
  64

  $ gitq 'commits take 3'
  99f91793  claude                2026-07-07  feat(gitq): .diff.lines morphism — matched diff content in the pipeline
  4df6371a  claude                2026-07-07  feat(gitq): prefill re-invoked query from a gitq-results-mode buffer
  bc019742  claude                2026-07-07  docs(gitq): write the coherent vision down — doc/gitq.org, README section

  $ gitq 'branches'
  main                                      99f91793
#+end_example

Other sources: =tags=, =refs=, =worktrees=, =blobs= (HEAD's files), =HEAD=,
or any ref/branch/SHA by name.

* Filtering with =where= (typed)

Conditions are validated against the fields the current frame actually
carries, and against each field's scalar type — see the error section at the
bottom for what happens when they don't fit.

#+begin_example
  $ gitq 'commits where message matches /^fix/ /count'
  22

  $ gitq 'commits where date within "1 week" /count'
  38

  $ gitq 'commits where author contains "claude", date within "2 days" /count'
  30

  $ gitq 'commits where parents-count > 1 /count'      # merge commits
  0
#+end_example

Comma chains conditions (AND).  Values: ="quoted strings"=, =/regexes/=
(with =matches=), bare numbers, bare words.  Date operators: =after=,
=before=, =within "N days/weeks/months/years"=.

* Revision ranges

#+begin_example
  $ gitq 'commits in main~5..main /count'
  5
#+end_example

Anything =git log= accepts as a range works: =main..feature=, =HEAD~10..=,
a single ref.

* Navigating the object graph with =via=

Morphism paths compose: each dotted segment is type-checked against what the
previous one produces.

#+begin_example
  $ gitq 'HEAD via .parent.parent'                     # grandparent
  bc019742  claude                2026-07-07  docs(gitq): write the coherent vision down — doc/gitq.org, README section

  $ gitq 'HEAD via .tree.entries[Blob] path "doc/*"'   # files under doc/ at HEAD
  doc/design.org
  doc/gitq.org

  $ gitq 'blobs path "bin/*" via .history take 3'      # commits that touched bin/
  99f91793  claude                2026-07-07  feat(gitq): .diff.lines morphism — matched diff content in the pipeline
  a2a6d2c7  claude                2026-07-07  feat(gitq): drop the leading "." on field names; validate against a closed list
  0de1c0e5  claude                2026-07-07  refactor(gitq): remove pipe syntax, gitq is now the flat-syntax command
#+end_example

Useful morphisms: =.parent= (also =.parent[N]=, =.parent*= ancestors,
=.parent†= children-of), =.tree=, =.tree.entries[Blob|Tree]=, =.diff=,
=.diff.hunks=, =.diff.lines=, =.history=, =.commit=.

* Searching commit messages and diffs

This is the section to know.  Three different questions, three different
tools — and they compose.

** "Which commits SAY something?" — =where message=

#+begin_example
  $ gitq 'commits where message contains "registry" take 3'
  d3f407c4  claude                2026-07-07  feat(gitq): terminal registry; /delete and /worktree actually work
  6c8a731e  claude                2026-07-07  feat(gitq): morphism registry with generic, type-checked composition
#+end_example

The results buffer shows each matching commit's subject line as the context.
For the full message *and* the full diff, press =RET= on a result — it opens
the commit in Magit (=magit-show-commit=).

** "What does a file CONTAIN at some commit?" — =grep=

#+begin_example
  $ gitq 'HEAD grep "defconst gitq--morphisms"'
  git-branch-off-gitq.el:1161: (defconst gitq--morphisms
#+end_example

=grep= searches the full blob content of every commit flowing into it and
returns one frame per matching line — =path:line: content= is the matched
line with its text, right in the results buffer.  Works on any commit set:
=commits take 20 grep /TODO|FIXME/= greps twenty commits' trees (one frame
per commit×line, so narrow the commit set first — it runs =git grep= once
per commit).  Patterns: ="literal"= or =/regex/=.

** "Which commits CHANGED something?" — =pickaxe=

#+begin_example
  $ gitq 'commits pickaxe "gitq--flat-parse-count" /count'
  1
#+end_example

=pickaxe= is git's =-S= (occurrence-count changed; add =regex= or use
=/.../ = for =-G=).  It keeps only the commits whose *diffs* added or
removed the pattern — but by itself it can't show you the matching lines…

** "Show me the matched diff lines" — =pickaxe= + =.diff.lines=

The flagship combination.  =.diff.lines= turns each commit into its actual
added/removed diff lines (text in =content=, ="+"/"-"= in =sign=), so after
=pickaxe= narrows the commits, a =content= filter leaves exactly the lines
that matched:

#+begin_example
  $ gitq 'commits pickaxe "gitq--flat-parse-count" via .diff.lines where content contains "gitq--flat-parse-count" sort path'
  1b84f860  git-branch-off-gitq.el:1069: +(defun gitq--flat-parse-count (tok step-name)
  1b84f860  git-branch-off-gitq.el:1117: +       (cons (list :type 'take :n (gitq--flat-parse-count (pop tokens) "take")) tokens))
  1b84f860  git-branch-off-gitq.el:1119: +       (cons (list :type 'skip :n (gitq--flat-parse-count (pop tokens) "skip")) tokens))
#+end_example

In the =*gitq*= buffer each line is rendered with =diff-added= /
=diff-removed= faces, prefixed by the short commit hash; =RET= jumps to that
commit's full diff in Magit for surrounding context.

Filter by direction with =sign= — only where the pattern was *introduced*:

#+begin_example
  $ gitq 'commits pickaxe "display-buffer-full-frame" via .diff.lines where content contains "display-buffer-full-frame", sign == "+" take 4'
  a684e8bd  git-branch-off-gitq.el:625: +`display-buffer-full-frame' does its work via `delete-other-windows',
  a684e8bd  git-branch-off-gitq.el:634: +    (display-buffer (gitq--render frames pipeline-str) '(display-buffer-full-frame))
  9d0b98c5  git-branch-off-gitq.el:615: +  (pop-to-buffer (gitq--render frames pipeline-str) '(display-buffer-full-frame)))
  9d0b98c5  git-branch-off-gitq.el:621: +\(display-buffer-full-frame\) rather than splitting it in half, without
#+end_example

(=sign == "-"= gives where it was removed; omit the =content= filter to see
a commit's whole diff as line frames.)

The =pickaxe= step is what keeps this fast: it prunes to the few relevant
commits *before* =.diff.lines= expands anything.  Running
=commits via .diff.lines ...= without narrowing first diffs every commit in
the repo.

** "Which FILES did matching commits change?" — =.diff= + =pick=

#+begin_example
  $ gitq 'commits where message matches /^fix/ take 2 via .diff pick path'
  :type:projection  :path:git-branch-off-gitq.el
  :type:projection  :path:test/git-branch-off-gitq-test.el
  :type:projection  :path:git-branch-off-gitq.el
  :type:projection  :path:test/git-branch-off-gitq-test.el
#+end_example

=.diff= yields one frame per changed path per commit; =pick= projects the
fields you care about (projections render as raw key:value pairs).

* The results buffer

In Emacs, =/show= (or no terminal at all) opens =*gitq*= full-frame:

| Key   | Action                                                        |
|-------+---------------------------------------------------------------|
| =RET= | Visit: commit → Magit diff view; blob → file at that revision |
| =b=   | Create a branch at the commit at point                        |
| =c=   | Copy the SHA at point                                         |
| =q=   | Quit                                                          |

Running =M-x gitq= again *from inside* a results buffer pre-fills the
minibuffer with the pipeline that produced it — tweak and re-run.

* Terminals in daily use

#+begin_example
  commits where message contains "tokenize" first /copy       ; SHA to kill-ring
  commits pickaxe "some-function" first /worktree             ; detached worktree at that commit
  commits where author contains "alice" first /branch-off "from-alice"
  HEAD via .parent* where message contains "fix" /count
#+end_example

=/worktree= places the worktree at =.worktree/<full-sha>/= (the package's
convention) unless you pass a quoted path.  =/delete= is an alias of
=/remove=; =/reword=, =/remove= cascade through the branch-off machinery.

* When a query is wrong, it says so

Nothing here returns a silent empty result — every mistake is a specific
parse-time error:

#+begin_example
  $ gitq 'commits where date > 2026'
  gitq: error: operator '>' does not apply to 'date' (a date field; try: ==, !=, after, before, within)

  $ gitq 'commits where name == "main"'
  gitq: error: field 'name' not valid here after 'where' (current frame has: sha, author, email, date, message, tree, parents-count)

  $ gitq 'HEAD via .tree.parent'
  gitq: error: 'via .tree.parent' needs a 'parents-count' field, but the current frame only has: sha

  $ gitq 'commits where message contains take'
  gitq: error: 'contains' requires a value; step keyword 'take' must be quoted: "take"
#+end_example

That last one matters when searching messages for words that are also gitq
keywords (=take=, =first=, =sort=, ...): quote them —
=commits where message contains "take"=.
